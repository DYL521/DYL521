I am not trying to make software look complicated.

I like building systems that make complexity disappear:

- search that can actually retrieve useful knowledge
- workflows that stop depending on manual chasing
- internal tools that keep saving time long after release

Most of my work sits somewhere between backend engineering, enterprise reality, and AI systems that need to be more than a demo.

### / how_i_got_here

I started from backend delivery, but the interesting part was never just writing APIs.

The real work was always around the system:

- who gets access
- what breaks when the process changes
- where information gets lost
- which manual step should not exist in the first place

That is why I ended up spending years around account lifecycle platforms, Jira and Confluence automation, and now AI knowledge systems.

  我不太喜欢把软件做得很“显复杂”。

  我更喜欢做另一种系统：

  - 让知识真正能被找到
  - 让流程不再依赖人工追着跑
  - 让内部工具上线之后还能持续替团队省时间

  我的工作一直在几个交叉点之间：后端工程、企业真实流程，以及必须真正落地的 AI 系统。

  <h3>/ 我怎么走到这里</h3>

  我最早是从后端开发做起，但后来发现，真正有意思的部分往往不只是接口和数据库。

  更麻烦、也更有价值的问题通常是：

  - 谁该有权限，谁不该有
  - 流程一变，系统哪里会跟着出问题
  - 信息是在哪个环节丢掉的
  - 哪个手工步骤其实根本不该存在

  所以我后来会长期做账号生命周期平台、Jira / Confluence 自动化，再到现在更专注 AI 知识系统，并不算跳跃，反而是一条很自然的线。

  <h3>/ 我在做什么</h3>

  <h4>RAG 与知识系统</h4>

  - 面向真实业务场景的 AI 后端，强调结构化检索、元数据过滤和可交付 API
  - 不是只做 Demo，而是做能进入组织流程的知识系统

  <h4>身份与流程平台</h4>

  - 覆盖入转调离的账号生命周期自动化
  - 打通 AD、人事系统、工单系统和内部业务平台

  <h4>Jira / Confluence 工程化</h4>

  - 深度使用 ScriptRunner 和 Groovy 做定制
  - 把审批、工作流、报表和流程控制做成可编排的内部能力
